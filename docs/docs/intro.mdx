# What is EnterpriseClaw?

EnterpriseClaw lets a company run its own AI assistant **inside a regulated corporate environment**. Your team gives it orders from Slack; it reasons with an LLM; and everything privileged it does flows through the GitOps controls the enterprise already trusts — Argo Events, Argo Workflows, a pull request, Argo CD, Crossplane. It is based on [OpenClaw](https://openclaw.ai/)'s AI assistant.

It is **not a chatbot**. It is an event-driven gateway for AI agent orchestration, built on two convictions:

> **AI proposes; a governed, auditable GitOps pipeline disposes.**
>
> **The security boundary is the service mesh — never the model.**

The agent can *reason* about what to build, but the only way anything gets created is a pull request a human merges. And whether the agent can reach any tool at all is decided by workload SPIFFE identity and Keycloak JWT claims enforced at the mesh — so even a fully prompt-injected agent cannot touch a tool its identity isn't allow-listed for.

## What you get

Running `enterpriseclaw init` (a Nushell CLI in a Devbox environment) takes you from zero to:

- an **AWS EKS** substrate provisioned with OpenTofu,
- an **Argo CD app-of-apps** reconciling the whole platform from your private config repo,
- an **Istio ambient** mesh with **agentgateway** as the L7 waypoint,
- the **kagent trio** (kagent + kmcp + agentgateway) running agents and MCP servers on-cluster,
- a **Slack front door** where each message becomes one short-lived, archived Argo Workflow run,
- and a **Session-Broker** integration binding each Slack user to a real corporate identity (Keycloak, Google-federated) — so privileged actions carry a *human* identity, not a bot token.
